# 19.5 — Nema Kernel (Multi-Threaded Foundation)

> **Nema** (Yunani νῆμα = "benang/thread") — nama kernel multi-thread Palanu, satu nuansa dengan Palanu (καιρός). Mengubah Palanu dari **cooperative single-loop** menjadi **preemptive multi-threaded** — terinspirasi Flipper Zero (Furi) tapi punya nama & desain sendiri, dan portable di dua target (ESP32 + simulator host). Ditulis jujur: tiap fase berdiri sendiri, bisa di-ship & dites, dan tidak mengklaim lebih dari yang dikerjakan.

> Catatan nama: "Furi" di dokumen ini SELALU merujuk sistem Flipper sebagai pembanding, bukan milik kita. Milik kita = **Nema**.

- Status: ⏳ Fase 0+1+TaskRunner DONE (build OK host+ESP32). Fase 2 di-rescope (lihat §13). Fase 1 belum diverifikasi di hardware oleh user saat dokumen ini ditulis.
- Milestone: M6.7 (Kernel Foundation)
- Depends on: 19 (AsyncEventPoster, AsyncDisplayDriver — sudah ada)
- Blocks: 20 (WiFi UI), app berat (download/HTTP), dan semua fitur masa depan yang butuh non-blocking

> **§13 — Revisi jujur saat implementasi (baca ini).** Fase 2 literal (pindahkan render loop ke thread sendiri) TIDAK jadi dikerjakan: display sudah async (display task) + input sudah async (input thread), jadi memindahkan render cuma menambah race surface (plugin di UI thread vs screen di GUI thread) tanpa melayani goal "jangan freeze". Sebagai gantinya goal itu dicapai lewat **`nema::TaskRunner`** (substansi asli Fase 3): kerja berat jalan di worker thread, hasil balik ke UI thread via callback yang di-drain `Runtime::step()`. UI tetap single-thread (aman, no race) tapi tidak pernah freeze. Dibuktikan: demo `TaskDemoPlugin` menjalankan "download" 3 detik sementara UI tetap render (17 frame tercatat selama job di simulator). Full per-app-thread (isolasi ala Flipper) ditunda — tidak diperlukan untuk 1 app foreground.

---

## 0. Kontrak Kejujuran (baca dulu)

Plan 19 sebelumnya **salah judul** — disebut "Non-Blocking Architecture" padahal cuma offload display + amankan event WiFi. Plan ini memperbaiki itu. Aturan dokumen ini:

1. **Tidak ada klaim "non-blocking" global.** FreeRTOS tidak otomatis non-blocking. Non-blocking = kerja berat **dipindah eksplisit** ke thread. Plan ini menyediakan **alat + memindahkan GUI/Input/App**, bukan sihir.
2. **Tiap fase harus tetap bikin sistem jalan** di simulator DAN hardware. Tidak ada big-bang rewrite.
3. **Apa yang TIDAK dikerjakan ditulis eksplisit** di tiap fase.
4. **Batas yang jujur**: simulator memvalidasi *logika & alur pesan*, BUKAN perilaku real-time. Timing/priority/race sebenarnya **hanya bisa diverifikasi di hardware**.

---

## 1. Bagaimana Flipper (Furi) Sebenarnya Bekerja

Furi = lapisan tipis di atas FreeRTOS. Yang relevan untuk kita (ini pembanding, bukan nama kita):

| Konsep Furi | Fungsi |
|---|---|
| `FuriThread` | wrapper FreeRTOS task — tiap app & service jalan di thread sendiri |
| `FuriMessageQueue` | queue thread-safe untuk kirim data antar-thread |
| `FuriPubSub` | publish/subscribe antar-thread |
| GUI service | thread sendiri, **memiliki** canvas + render loop; app kirim ViewPort update |
| Input service | thread sendiri, baca GPIO → kirim `InputEvent` ke queue |
| `furi_record_*` | service registry (kita: ServiceContainer) |
| ViewDispatcher + SceneManager | navigasi layar (kita: ViewDispatcher) |

**Inti yang membedakan dari Palanu sekarang:** di Furi, app boleh blocking (nunggu queue, download) tanpa membekukan GUI/Input — karena scheduler preempt thread lain. Palanu sekarang: satu plugin blocking = semua beku.

**Realisme Furi (penting):** Flipper TIDAK punya ratusan thread. Pola nyatanya: **beberapa service-thread persisten** (gui, input, storage, notification) **+ 1 app foreground thread**. Kita ikut model ini.

- Input sebagai thread (sekarang poll di main loop → bug loncat).
- Model app per-thread.

## 3. Batasan Desain #1: Dual-Build (yang Flipper TIDAK punya)

Flipper hardware-only. Palanu **wajib jalan di simulator host juga**. Maka tiap primitif Nema butuh **dua implementasi di balik satu interface**:

| Primitif | Impl ESP32 | Impl Host |
|---|---|---|
| `nema::Thread` | `xTaskCreatePinnedToCore` (kontrol stack/core/prio) | `std::thread` |
| `nema::MessageQueue<T>` | mutex + deque + counting semaphore | mutex + deque + condvar |
| `Mutex` | `std::mutex` (sudah terbukti jalan di ESP-IDF) | `std::mutex` |

**Keputusan jujur soal `std::thread`:** ESP-IDF *mendukung* `std::thread`, TAPI stack size / core / priority hanya bisa diset global lewat menuconfig — tidak per-thread. Untuk kernel embedded kita butuh kontrol eksplisit per-thread, jadi **di ESP32 pakai FreeRTOS langsung**, di host pakai `std::thread`. Bukan "std::thread di mana-mana".

**Keputusan jujur soal `MessageQueue<T>`:** payload kita (`Event`) mengandung `std::string` (non-POD). FreeRTOS `xQueue` menyalin byte mentah → rusak untuk non-POD. Maka `nema::MessageQueue<T>` generik pakai **mutex + deque** (pola yang sudah terbukti di `AsyncEventPoster`) di kedua platform. FreeRTOS native `xQueue` hanya dipakai untuk jalur **POD + ISR** (input dari interrupt) — lihat Fase 1.

## 4. Primitif Inti (`firmware/core/include/palanu/nema/`)

Semua primitif kernel hidup di namespace `nema::nema` dan direktori `palanu/nema/`.

### 4.1 `Thread`

```cpp
namespace nema::nema {

struct ThreadConfig {
    const char* name       = "thread";
    uint32_t    stackBytes = 4096;   // ESP32: stack task. Host: diabaikan.
    uint8_t     priority   = 5;      // ESP32 FreeRTOS prio. Host: diabaikan.
    int8_t      core       = -1;     // ESP32: 0/1, -1=no affinity. Host: diabaikan.
};

class Thread {
public:
    using Entry = void(*)(void*);
    Thread(const ThreadConfig& cfg, Entry entry, void* arg);
    ~Thread();              // memastikan thread berhenti & resource dilepas
    void requestStop();     // set flag; entry harus cek shouldStop()
    bool shouldStop() const;
    void join();            // tunggu selesai (host: join; ESP32: completion sem)
private:
    // ESP32: TaskHandle_t + SemaphoreHandle_t (completion)
    // Host:  std::thread
};

} // namespace nema::nema
```

**Catatan jujur:** FreeRTOS tidak punya `join` native. Kita implement via *completion semaphore* yang di-`give` saat entry selesai, lalu `join()` mem-`take`. Entry **wajib** loop `while (!shouldStop())` — tidak ada cara aman "kill" thread yang lagi pegang lock.

### 4.2 `MessageQueue<T>`

```cpp
namespace nema::nema {

template <typename T>
class MessageQueue {
public:
    explicit MessageQueue(size_t capacity);
    bool send(T item, uint32_t timeoutMs = 0);   // false jika penuh/timeout
    bool receive(T& out, uint32_t timeoutMs);    // false jika kosong/timeout
    size_t size() const;
private:
    std::deque<T> q_;
    std::mutex    m_;
    // ESP32: counting semaphore untuk blocking receive
    // Host:  std::condition_variable
};

} // namespace nema::nema
```

`AsyncEventPoster` yang sekarang menjadi kasus khusus: `nema::MessageQueue<Event>` + helper `flush(EventBus&)`. Kita refactor poster agar memakai `MessageQueue` di dalamnya (tidak menulis ulang mekanisme).

### 4.3 `Mutex`
`std::mutex` langsung (sudah terbukti jalan di ESP32 — Logger memakainya). Tidak perlu wrapper. Aturan: **jangan pernah pegang mutex saat memanggil `send`/`receive` queue** (cegah deadlock).

---

## 5. Aturan Konkurensi (wajib, biar tidak jadi mimpi buruk race)

Race/deadlock adalah harga multi-threading. Furi matang setelah bertahun-tahun. Kita disiplin dari awal:

1. **Message-passing > shared memory.** Antar-thread komunikasi lewat `nema::MessageQueue`, bukan variabel bersama.
2. **Single-owner.** Tiap data punya SATU thread pemilik yang boleh menulis. Contoh: Canvas dimiliki GUI thread saja.
3. **`EventBus` tetap single-thread.** Publish lintas-thread DILARANG langsung — wajib lewat queue/poster yang di-drain oleh thread pemilik EventBus. Tambah assertion debug: `assert(currentThread == ownerThread)` di `EventBus::publish`.
4. **Jangan pegang lock saat I/O / queue op.** Lock hanya untuk swap/copy singkat (pola yang sudah dipakai AsyncDisplayDriver & AsyncEventPoster).
5. **Stack di internal SRAM, bukan PSRAM.** Stack task harus cepat & ISR-safe. Buffer besar (framebuffer) boleh PSRAM.

## 6. Fase Implementasi (tiap fase shippable & testable)

### Fase 0 — Primitif + test (NOL perubahan perilaku)
- Buat `palanu/nema/thread.h` (+ impl esp32 & host), `palanu/nema/message_queue.h`.
- Refactor `AsyncEventPoster` agar pakai `nema::MessageQueue<Event>` (perilaku identik).
- Unit test di host: spawn thread, kirim/terima pesan, join.
- **TIDAK** menyentuh main loop, GUI, input. Sistem jalan persis seperti sekarang.
- **Selesai bila:** simulator + ESP32 build & jalan tanpa perubahan tampak; test thread/queue lulus di host.

### Fase 1 — Input thread (fix bug loncat, risiko rendah, terisolasi)
- `InputService` jadi thread sendiri: poll TCA9534 (atau pakai IRQ `PIN_BTN_IRQ` → ISR → native POD queue) → kirim `InputEvent{key, edge}` ke `nema::MessageQueue<InputEvent>`.
- Main loop drain input queue → `viewDispatcher.handleKey()` (sementara masih di main loop).
- **Kenapa ini dulu:** menuntaskan masalah "pencet hilang/loncat" secara benar (input tak pernah hilang karena polling tak lagi terhambat apa pun), DAN membuktikan primitif Fase 0 di hardware.
- **TIDAK** memindahkan GUI dulu. Main loop masih ada.
- **Selesai bila:** di hardware, navigasi cepat tidak pernah skip/hilang; tekan saat e-ink refresh tetap tercatat.

### Fase 2 — GUI service-thread
- `GuiService` jadi thread sendiri: **memiliki** Canvas + ViewDispatcher + render loop.
- App/plugin kirim "redraw request" / view-model ke GUI lewat queue; tidak pernah menyentuh Canvas langsung.
- `AsyncDisplayDriver` tetap jadi layer panel I/O di bawah GUI thread (flush SPI). Jadi dua lapis: GUI thread (gambar ke buffer) → display task (SPI ke kaca).
- Input queue sekarang dikonsumsi GUI thread (bukan main loop).
- **TIDAK** mengubah cara plugin ditulis (API IScreen tetap) selain larangan akses Canvas lintas-thread.
- **Selesai bila:** render & input jalan tanpa main loop; UI tetap identik; tidak ada race pada Canvas (single-owner).

### Fase 3 — App threading model
- App foreground aktif jalan di **thread sendiri** (spawn saat launch, stop saat exit). Pola Flipper: 1 app foreground + service-thread persisten.
- App boleh blocking (mis. `download()`) tanpa membekukan GUI/Input.
- `PluginContext` dapat akses: kirim view update ke GUI, subscribe event lewat queue.
- Background job (download/HTTP/file) jalan di dalam thread app, hasil balik via queue.
- **TIDAK** bikin tiap plugin pasif jadi thread — hanya app *aktif/foreground*. Plugin ringan event-driven boleh tetap di service tick.
- **Selesai bila:** sebuah app contoh melakukan operasi blocking lama (simulasi download dengan `sleep`) dan UI/tombol tetap responsif.

## 7. Anggaran Memori (jujur, harus dilacak)

Stack thread di **internal SRAM** (ESP32-S3: 512KB). Estimasi:

| Thread | Stack |
|---|---|
| GUI service | ~8 KB |
| Input service | ~2 KB |
| App foreground | ~8 KB |
| Display task (sudah ada) | ~8 KB |
| **Total tambahan** | **~26 KB** |

Aman di 512KB, tapi **wajib dilacak** seiring nambah service. WiFi/TLS bisa makan 50KB+. Framebuffer (3×46KB) sudah di PSRAM. Aturan: stack jangan di PSRAM.

## 8. Risiko & Mitigasi (jujur)

| Risiko | Mitigasi |
|---|---|
| Race condition | Aturan §5 (message-passing, single-owner), assertion thread-owner di EventBus |
| Deadlock | Larangan pegang lock saat queue/I-O op |
| Priority inversion | Prioritas eksplisit; GUI > app; input tinggi; hindari lock antar-prioritas |
| Stack overflow | Stack size eksplisit + `uxTaskGetStackHighWaterMark` saat dev |
| Simulator ≠ real-time | Diakui: host validasi logika/alur, bukan timing. Verifikasi akhir di hardware. |
| Regresi besar | Fase kecil; tiap fase build & jalan di kedua target |

## 9. Non-Goals (eksplisit — biar tidak over-promise)

Plan ini **TIDAK** mengerjakan:
- Clone Furi penuh (tanpa `FuriEventLoop`-equivalent, SceneManager, pubsub lintas-thread lengkap di v1).
- Memory protection / sandbox antar-app (ESP32 tanpa MMU app-isolation).
- Menjadikan SETIAP plugin pasif sebuah thread.
- Menghapus main loop sepenuhnya di Fase 1.
- Membuat operasi apa pun "otomatis" non-blocking — hanya menyediakan alat + memindahkan GUI/Input/App.
- OTA, scheduler hemat daya, tickless idle (plan terpisah).
