# 19.6 — Nema App Model (Full Furi Parity & Beyond)

> Model penuh ala Furi: GUI service-thread + app per-thread + message-passing, supaya app boleh blocking/crash tanpa menjatuhkan sistem. PLUS desain yang memanfaatkan keunggulan ESP32-S3 untuk **melampaui** Flipper. Ditulis jujur, fase kecil yang tiap langkah build & jalan.

- Status: 🚧 Fase A ✅ (HW) · Fase B ✅ (Counter app-thread, HW) · Fase C ✅ build host+ESP32 (Clock/Stopwatch/TaskDemo → apps, status bar in apps); belum diverifikasi HW · Fase D belum
- Milestone: M6.8 (App Model)

> **Catatan eksekusi Fase C (jujur):** (1) Status bar kini tampil di app — `IApp::fullscreen()` default false → AppHost mode = Normal, GuiService gambar status bar lalu AppHost blit frame app mulai di bawah separator (`SEP1_Y+2`). Stopwatch `fullscreen()==true`. (2) Clock/Stopwatch/TaskDemo dimigrasi jadi `IApp` (ClockApp/StopwatchApp/TaskDemoApp), plugin jadi thin launcher. TaskDemoApp pakai TaskRunner + `std::atomic` flag (worker set, app thread poll) → app tetap responsif & cancelable saat job 3s jalan. (3) **Keputusan jujur:** Settings/About/Logs/Home TETAP cooperative system-screen, BUKAN app-thread — screen ini tidak pernah blocking, jadi spawn thread = overhead murni tanpa nilai paritas. Helper `ui::drawTitle()` menyeragamkan padding header semua screen. Verifikasi host: 205 frame, 0 JSON corruption, 3 app load app-threaded.

> **Catatan eksekusi Fase B (jujur):** protokol app↔GUI memakai **buffer handoff**, bukan "render-command list" seperti draft awal §3. Alasan: app menggambar dengan API Canvas yang sama (BufferDisplay → buffer RAM), lalu `present()` mem-copy buffer ke slot yang dibaca GUI thread di bawah mutex. Hasilnya: reuse penuh kode draw, dan **satu-satunya state lintas-thread = pixel buffer + input mailbox** (dua-duanya terkunci) → no shared model, no race by construction. `AppHost` (IScreen + AppContext) menjembatani app-thread ke ViewDispatcher tanpa GuiService perlu tahu soal app. Counter dimigrasi sebagai `CounterApp` percontohan. Status bar di app-mode = Fase C (sekarang app Fullscreen).
- Depends on: 19.5 (nema::Thread, MessageQueue, TaskRunner — DONE & verified hardware)
- Blocks: app store / OTA / networked apps

> Kontrak kejujuran sama seperti 19.5: tidak ada klaim ajaib; tiap fase shippable; non-goals eksplisit; race ditangani by design (message-passing, single-owner); verifikasi akhir di hardware.

## 1. Analisis Kompetitif (kenapa model penuh layak)

### Flipper Furi (target paritas)
STM32WB55: 1 core app M4 @64MHz + M0+ (radio BLE saja), 256KB RAM, 1MB flash, 128×64.
- `FuriThread` (app & service = thread), `FuriMessageQueue`, `FuriPubSub`, `FuriEventLoop`, `FuriTimer`
- GUI service (punya canvas, ViewPort/View/ViewDispatcher), Input service, Storage, Notification
- `furi_record_*` (service registry), SceneManager (navigasi)
- App di-launch di thread sendiri; loader kelola lifecycle; 1 app foreground + service

### Palanu ESP32-S3 (kita)
2 core LX7 @240MHz (dua-duanya bebas), 512KB SRAM + 8MB PSRAM, 8MB flash, 264×176.

### Sudah setara (built di 19.5)
`nema::Thread` ↔ FuriThread · `nema::MessageQueue` ↔ FuriMessageQueue · `TaskRunner` ↔ thread pool · `ServiceContainer` ↔ record · `ViewDispatcher` ↔ ViewDispatcher · `AsyncEventPoster` ↔ pub/sub parsial.

### Belum ada (gap → dikerjakan di sini)
GUI-thread · App-thread (`IApp`) · message-passing UI · AppLoader/lifecycle · SceneManager.

### Di mana Palanu MELAMPAUI Flipper (jadikan fitur unggulan)
| Dimensi | Flipper | Palanu | Keunggulan |
|---|---|---|---|
| Core | 1 @64MHz | 2 @240MHz | GUI core-1, app/net core-0 → paralel sejati |
| RAM | 256KB | 512KB + 8MB PSRAM | app heap besar di PSRAM |
| Net | — | WiFi + BLE | networked app, OTA, remote, cloud |
| Flash | 1MB | 8MB | 8× app & aset |
| Dev | hardware-only | dual-build simulator + web UI | dev/tes/CI tanpa hardware |
| Portabilitas | terikat HW | HAL + capability | app jalan sim→devboard→V1 |

**Tiga pilar unggulan yang harus dipanggang ke arsitektur:** (1) paralelisme dual-core, (2) platform app ber-jaringan, (3) simulator-first.

## 2. Arsitektur Target

```
            ┌──────────── core 1 (UI) ────────────┐   ┌──── core 0 (work/net) ────┐
            │  GuiService (thread)                 │   │  TaskRunner worker(s)      │
            │   owns: Canvas, ViewDispatcher       │   │  WiFi/net, download, crypto│
            │   loop: drain input → route to app   │   │                            │
            │         drain render-cmds → draw      │   └────────────────────────────┘
            │         flush → AsyncDisplayDriver    │   ┌──── core 0 (input) ───────┐
            └──────────────▲───────────┬───────────┘   │  Input poll thread         │
              render-cmd /  │           │ input-event   └──────────┬─────────────────┘
              view-model    │           ▼                          │ InputService queue
            ┌───────────────┴───────────────────────┐             │
            │  App thread (foreground)               │◄────────────┘ (GUI routes to focused app)
            │   IApp::run(AppContext&)               │
            │   may BLOCK (download) — only this app  │
            │   sends view-model/render-cmd to GUI    │
            │   receives input from its mailbox       │
            └─────────────────────────────────────────┘
```

**Prinsip kunci (hilangkan race by design):**
- **GUI thread satu-satunya pemilik Canvas + ViewDispatcher.** Tidak ada thread lain menyentuhnya.
- **App tidak pernah menyentuh Canvas.** App mengirim *view-model* atau *render-command* ke GUI via queue.
- **Input**: poll thread → InputService → GUI thread → diteruskan ke mailbox app yang fokus.
- **Hasil kerja berat**: TaskRunner → callback di thread app (atau via mailbox), bukan di GUI.
- Tidak ada shared mutable state lintas-thread; semua lewat message. Inilah yang membuat Fase ini aman padahal multi-thread.

## 3. Komponen Baru

### 3.1 `IApp` + `AppContext`
```cpp
struct IApp {
    virtual const char* id()   const = 0;
    virtual const char* name() const = 0;
    virtual void run(AppContext& ctx) = 0;   // jalan di thread app; boleh blocking
};
```
`AppContext` memberi app: kirim view ke GUI (`ctx.gui().present(view)`), terima input (`ctx.input().receive(ev, timeout)`), akses TaskRunner, log, events, capabilities, `ctx.shouldExit()`.

App loop tipikal:
```cpp
void MyApp::run(AppContext& ctx) {
    Model m;
    while (!ctx.shouldExit()) {
        InputEvent ev;
        if (ctx.input().receive(ev, 50)) m.handle(ev);   // non-blocking-ish
        ctx.gui().present(render(m));                     // kirim view-model ke GUI
    }
}
```

### 3.2 `GuiService` (thread, core 1)
- Owns Canvas + ViewDispatcher.
- Loop: drain InputService → route ke app fokus; drain render-queue → gambar view-model → flush.
- Status bar / modal tetap di GUI (sistem), digambar sebelum view app.

### 3.3 `AppLoader`
- `launch(IApp&)` → spawn `nema::Thread`, set jadi app fokus, kirim back-stack.
- `exit()` → sinyal `shouldExit`, join thread, kembali ke app sebelumnya (atau home).
- 1 app foreground + service persisten (pola Flipper). Stack app dari internal SRAM, heap app dari PSRAM.

### 3.4 View-model / render-command protocol
Dua opsi (putuskan saat Fase 2):
- **A. View-model**: app kirim struct data (mis. daftar item + cursor), GUI yang tahu cara menggambarnya. Aman, tapi GUI harus tahu tiap tipe view.
- **B. Render-command list**: app kirim daftar primitif (`drawText`, `rect`, `invertRect`). Fleksibel (app gambar apa saja) tanpa GUI tahu tipenya. Lebih dekat ke Furi Canvas-callback.
Rekomendasi: **B** (render-command) — fleksibel, app tak terbatas tipe view bawaan, dan tetap aman karena command di-replay GUI thread di canvas miliknya.

## 4. Fase (tiap fase build + jalan di host & ESP32)

### Fase A — GuiService thread (cutover render)
- Pindahkan blok render + input-drain + viewDispatcher.tick dari `Runtime::step()` ke `GuiService` thread (core 1). Runtime main loop tinggal: service tick, plugin tick, asyncPoster/task drain.
- **Sementara** plugin lama tetap jalan; karena belum message-passing, GUI thread dan plugin tick bisa sentuh state sama → kita kunci interim: plugin tick & GUI render di-serialize lewat 1 mutex UI **atau** (lebih baik) langsung lanjut Fase B agar tidak ada shared state. Catatan jujur: Fase A sendirian tidak aman tanpa B.
- **Selesai bila:** render jalan di GuiService thread, UI identik, sim+ESP32 OK.

### Fase B — IApp + message-passing + migrasi 1 app
- Definisikan `IApp`/`AppContext` + render-command queue + input mailbox.
- Migrasi SATU app (mis. Counter) jadi `IApp` murni: jalan di thread sendiri, kirim render-command, terima input. Tidak menyentuh Canvas.
- AppLoader launch/exit untuk app itu.
- **Selesai bila:** Counter jalan sebagai app-thread, race-free (Canvas single-owner GUI), bisa di-launch & exit bersih.

### Fase C — Migrasi semua + hapus model plugin lama UI
- Clock, Stopwatch, TaskDemo, screens sistem (Settings/About/Logs/WiFi) → IApp / system views.
- Home/launcher jadi system app yang me-launch app lain.
- Plugin "pasif" (tanpa UI, mis. background sensor) tetap service.
- **Selesai bila:** semua UI lewat app-model; tidak ada lagi screen yang mutate dari main loop.

### Fase D — Keunggulan di atas Flipper (pilar unggulan)
- **Networked app**: contoh app yang fetch data via WiFi (pakai TaskRunner) → tampil — sesuatu yang Flipper tak bisa.
- **Per-core tuning**: pastikan GUI core-1, net/worker core-0; ukur paralelisme.
- **Simulator parity**: app yang sama jalan di web simulator + hardware; jadikan jalur dev resmi.
- (OTA & app-store = plan terpisah, tapi pondasinya di sini.)

## 5. Risiko & Mitigasi (jujur)
| Risiko | Mitigasi |
|---|---|
| Race UI | Canvas single-owner GUI; app via message saja; no shared mutable UI state |
| Deadlock | Tidak pegang lock saat queue op; mailbox bounded + timeout |
| Priority inversion | GUI prio > app; input tinggi; net/worker rendah |
| Memori (stack tiap thread) | Stack internal SRAM, app heap PSRAM; lacak high-water-mark; batasi jumlah thread (1 app foreground) |
| Refactor besar memecah yang jalan | Fase kecil; tiap fase build+jalan; migrasi 1 app dulu (Fase B) sebelum semua |
| Fase A tak aman sendirian | Akui: A+B dikerjakan berdekatan; jangan ship A tanpa B |
| Sim ≠ real-time | Host validasi logika/alur; timing final di hardware |

---

## 6. Non-Goals (eksplisit)
- Memory protection/sandbox antar-app (ESP32 tanpa MMU app-isolation) — kita dapat *crash isolation* via thread, bukan memory isolation.
- Multi app foreground bersamaan (tetap 1 foreground; multitasking via service/background job).
- App store / OTA implementasi (pondasi saja di sini).
- FuriEventLoop/Timer lengkap (pakai pola loop + MessageQueue dulu).
